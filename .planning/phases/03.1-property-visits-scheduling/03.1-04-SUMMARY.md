## API Endpoints Created

### Availability (Landlord)
- `POST /visits/properties/:propertyId/availability` - Configure availability (VISIT-01)
- `GET /visits/properties/:propertyId/availability` - Get availability (public)
- `PATCH /visits/properties/:propertyId/availability/:availabilityId` - Update
- `DELETE /visits/properties/:propertyId/availability/:availabilityId` - Delete

### Slots (Public)
- `GET /visits/properties/:propertyId/slots` - Get available slots (VISIT-03, VISIT-05)

### Visit Lifecycle (Tenant)
- `POST /visits` - Request visit (VISIT-04)
- `GET /visits/mine` - My visits (VISIT-11)
- `POST /visits/:id/reschedule` - Reschedule visit (VISIT-08)

### Landlord
- `GET /visits/landlord` - All visits for my properties (VISIT-12)
- `GET /visits/properties/:propertyId` - Visits for specific property
- `PATCH /visits/:id/accept` - Accept request (VISIT-06)
- `PATCH /visits/:id/reject` - Reject with reason (VISIT-07)
- `PATCH /visits/:id/complete` - Mark completed

### Shared
- `GET /visits/:id` - Get visit details
- `PATCH /visits/:id/cancel` - Cancel with reason (VISIT-09)

## Decisions Made

1. **Required reason for reject/cancel (10-500 chars)** - Transparency for both parties
2. **Reschedule creates new visit** - Original marked RESCHEDULED, new visit PENDING with rescheduledFromId link
3. **Either party can reschedule accepted visits** - State machine allows both TENANT and LANDLORD to reschedule ACCEPTED status
4. **All status changes emit events** - visit.statusChanged event for Phase 13 notification integration
