# Phase 3.1: Property Visits Scheduling - Research

**Researched:** 2026-02-02
**Domain:** Appointment scheduling, availability management, time slot generation
**Confidence:** HIGH

## Summary

Phase 3.1 implements property visit scheduling for the Colombian rental market. Landlords configure their availability (weekly recurring schedule with day/hours and slot duration), and tenants can view available slots, request visits, and manage their scheduled appointments. The system must prevent double-booking, support a clear status workflow (PENDING, ACCEPTED, REJECTED, CANCELLED, COMPLETED, RESCHEDULED), and prepare notification hooks for Phase 13.

The implementation requires four main components:
1. **PropertyAvailability model** - Stores landlord's weekly recurring schedule (day of week + time windows)
2. **PropertyVisit model** - Individual visit requests with status tracking
3. **Slot generation service** - Generates available slots from availability config, filtering out booked slots
4. **Visit state machine** - Manages status transitions with role-based permissions

**Primary recommendation:** Store availability as recurring weekly rules (day of week + start/end times + slot duration), generate slots on-demand when tenant queries, and use database transactions with unique constraints to prevent double-booking. Use Colombia timezone (America/Bogota, UTC-5) for all date/time handling.

### Pattern 2: On-Demand Slot Generation
**What:** Generate available slots dynamically when tenant requests
**When to use:** Tenant views available slots for a date range
**Example:**
```typescript
// slots.service.ts
import { TZDate } from '@date-fns/tz';
import {
  addMinutes,
  format,
  startOfDay,
  addDays,
  getDay,
  isBefore,
  isAfter
} from 'date-fns';

const COLOMBIA_TZ = 'America/Bogota';

interface TimeSlot {
  date: string;        // "2026-02-15"
  startTime: string;   // "09:00"
  endTime: string;     // "09:30"
  isAvailable: boolean;
}

async generateSlotsForDateRange(
  propertyId: string,
  startDate: Date,
  endDate: Date,
): Promise<TimeSlot[]> {
  // 1. Get availability rules for property
  const availabilities = await this.prisma.propertyAvailability.findMany({
    where: { propertyId, isActive: true },
  });

  // 2. Get existing visits (PENDING or ACCEPTED) in date range
  const bookedVisits = await this.prisma.propertyVisit.findMany({
    where: {
      propertyId,
      visitDate: { gte: startDate, lte: endDate },
      status: { in: ['PENDING', 'ACCEPTED'] },
    },
  });

  // 3. Build set of booked slots for O(1) lookup
  const bookedSlots = new Set(
    bookedVisits.map(v => `${format(v.visitDate, 'yyyy-MM-dd')}_${v.startTime}`)
  );

  const slots: TimeSlot[] = [];
  let currentDate = startOfDay(startDate);

  // 4. Iterate through each day in range
  while (isBefore(currentDate, endDate) || format(currentDate, 'yyyy-MM-dd') === format(endDate, 'yyyy-MM-dd')) {
    const dayOfWeek = getDay(currentDate); // 0-6

    // 5. Find availability rules for this day
    const dayAvailabilities = availabilities.filter(a => a.dayOfWeek === dayOfWeek);

    for (const availability of dayAvailabilities) {
      // 6. Generate slots within this availability window
      let slotStart = this.parseTime(availability.startTime);
      const windowEnd = this.parseTime(availability.endTime);

      while (slotStart + availability.slotDuration <= windowEnd) {
        const slotStartStr = this.formatMinutes(slotStart);
        const slotEndStr = this.formatMinutes(slotStart + availability.slotDuration);
        const dateStr = format(currentDate, 'yyyy-MM-dd');
        const slotKey = `${dateStr}_${slotStartStr}`;

        slots.push({
          date: dateStr,
          startTime: slotStartStr,
          endTime: slotEndStr,
          isAvailable: !bookedSlots.has(slotKey),
        });

        slotStart += availability.slotDuration;
      }
    }

    currentDate = addDays(currentDate, 1);
  }

  return slots;
}

// Helper: parse "09:30" to minutes since midnight (570)
private parseTime(time: string): number {
  const [hours, minutes] = time.split(':').map(Number);
  return hours * 60 + minutes;
}

// Helper: format minutes since midnight to "09:30"
private formatMinutes(minutes: number): string {
  const h = Math.floor(minutes / 60);
  const m = minutes % 60;
  return `${h.toString().padStart(2, '0')}:${m.toString().padStart(2, '0')}`;
}
```

### Pattern 3: Visit Status State Machine
**What:** Explicit state transitions with role-based permissions
**When to use:** All visit status changes
**Example:**
```typescript
// visit-state-machine.ts
import { Injectable, BadRequestException, ForbiddenException } from '@nestjs/common';

export enum VisitStatus {
  PENDING = 'PENDING',
  ACCEPTED = 'ACCEPTED',
  REJECTED = 'REJECTED',
  CANCELLED = 'CANCELLED',
  COMPLETED = 'COMPLETED',
  RESCHEDULED = 'RESCHEDULED',
}

type ActorRole = 'TENANT' | 'LANDLORD';

interface Transition {
  from: VisitStatus;
  to: VisitStatus;
  allowedRoles: ActorRole[];
}

@Injectable()
export class VisitStateMachine {
  private readonly transitions: Transition[] = [
    // Landlord accepts/rejects pending visits
    { from: VisitStatus.PENDING, to: VisitStatus.ACCEPTED, allowedRoles: ['LANDLORD'] },
    { from: VisitStatus.PENDING, to: VisitStatus.REJECTED, allowedRoles: ['LANDLORD'] },

    // Either party can cancel pending
    { from: VisitStatus.PENDING, to: VisitStatus.CANCELLED, allowedRoles: ['TENANT', 'LANDLORD'] },

    // Tenant can reschedule pending (creates new visit)
    { from: VisitStatus.PENDING, to: VisitStatus.RESCHEDULED, allowedRoles: ['TENANT'] },

    // Either party can cancel accepted
    { from: VisitStatus.ACCEPTED, to: VisitStatus.CANCELLED, allowedRoles: ['TENANT', 'LANDLORD'] },

    // Landlord marks visit as completed
    { from: VisitStatus.ACCEPTED, to: VisitStatus.COMPLETED, allowedRoles: ['LANDLORD'] },

    // Accepted can be rescheduled (needs landlord approval -> new PENDING)
    { from: VisitStatus.ACCEPTED, to: VisitStatus.RESCHEDULED, allowedRoles: ['TENANT', 'LANDLORD'] },
  ];

  canTransition(from: VisitStatus, to: VisitStatus, role: ActorRole): boolean {
    return this.transitions.some(
      t => t.from === from && t.to === to && t.allowedRoles.includes(role)
    );
  }

  validateTransition(from: VisitStatus, to: VisitStatus, role: ActorRole): void {
    const transition = this.transitions.find(t => t.from === from && t.to === to);

    if (!transition) {
      throw new BadRequestException(
        `Invalid transition from ${from} to ${to}`,
      );
    }

    if (!transition.allowedRoles.includes(role)) {
      throw new ForbiddenException(
        `Role ${role} cannot transition visit from ${from} to ${to}`,
      );
    }
  }

  getAvailableTransitions(from: VisitStatus, role: ActorRole): VisitStatus[] {
    return this.transitions
      .filter(t => t.from === from && t.allowedRoles.includes(role))
      .map(t => t.to);
  }

  isTerminal(status: VisitStatus): boolean {
    return [VisitStatus.COMPLETED, VisitStatus.REJECTED, VisitStatus.CANCELLED].includes(status);
  }
}
```

### Pattern 4: Double-Booking Prevention
**What:** Database-level constraint + application transaction
**When to use:** Creating or rescheduling visits
**Example:**
```typescript
// Prisma schema: Unique constraint prevents same property+date+time
model PropertyVisit {
  // ...
  @@unique([propertyId, visitDate, startTime, status], name: "unique_active_booking")
  // Note: This partial unique won't work in Prisma - use application logic
}

// visits.service.ts - Application-level check with transaction
async createVisit(
  tenantId: string,
  dto: CreateVisitDto,
): Promise<PropertyVisit> {
  return this.prisma.$transaction(async (tx) => {
    // 1. Check if slot is still available (with row lock)
    const existingVisit = await tx.propertyVisit.findFirst({
      where: {
        propertyId: dto.propertyId,
        visitDate: new Date(dto.date),
        startTime: dto.startTime,
        status: { in: ['PENDING', 'ACCEPTED'] },
      },
    });

    if (existingVisit) {
      throw new ConflictException('This time slot is no longer available');
    }

    // 2. Verify slot exists in availability config
    const dayOfWeek = getDay(new Date(dto.date));
    const availability = await tx.propertyAvailability.findFirst({
      where: {
        propertyId: dto.propertyId,
        dayOfWeek,
        isActive: true,
      },
    });

    if (!availability) {
      throw new BadRequestException('Property is not available on this day');
    }

    // Verify time is within availability window
    const slotMinutes = this.parseTime(dto.startTime);
    const windowStart = this.parseTime(availability.startTime);
    const windowEnd = this.parseTime(availability.endTime);

    if (slotMinutes < windowStart || slotMinutes + availability.slotDuration > windowEnd) {
      throw new BadRequestException('Selected time is outside availability window');
    }

    // 3. Create the visit
    const visit = await tx.propertyVisit.create({
      data: {
        propertyId: dto.propertyId,
        tenantId,
        visitDate: new Date(dto.date),
        startTime: dto.startTime,
        endTime: this.formatMinutes(slotMinutes + availability.slotDuration),
        status: VisitStatus.PENDING,
        notes: dto.notes,
      },
    });

    // 4. Emit event for notifications (Phase 13)
    this.eventEmitter.emit('visit.requested', new VisitRequestedEvent(visit));

    return visit;
  });
}
```

### Pattern 5: Timezone Handling (Colombia)
**What:** Use consistent timezone for all date/time operations
**When to use:** Any date/time calculation or display
**Example:**
```typescript
import { TZDate } from '@date-fns/tz';
import { format, startOfDay, addDays } from 'date-fns';

const COLOMBIA_TZ = 'America/Bogota'; // UTC-5, no DST

// Creating a date in Colombia timezone
const colombiaDate = new TZDate(2026, 1, 15, 9, 0, COLOMBIA_TZ); // Feb 15, 2026 at 9:00 AM

// Converting UTC to Colombia time for display
function toColombiaTime(utcDate: Date): TZDate {
  return new TZDate(utcDate, COLOMBIA_TZ);
}

// Formatting for API responses (ISO with timezone)
function formatForResponse(date: TZDate): string {
  return format(date, "yyyy-MM-dd'T'HH:mm:ssXXX"); // "2026-02-15T09:00:00-05:00"
}

// Important: Store dates in UTC in database, convert for display
// Prisma stores DateTime as UTC by default
```

### Anti-Patterns to Avoid
- **Pre-generating all slots:** Don't create slot records in database - generate on demand
- **Storing local times without timezone:** Always use UTC in database, convert for display
- **Trusting client-side date validation:** Validate all dates server-side
- **Updating availability without checking existing visits:** Warn or prevent if visits exist
- **Not handling timezone in slot comparison:** Colombia is UTC-5 with no DST
- **Missing transactions for booking:** Race conditions will cause double-booking

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Date arithmetic | Manual day/month/year math | date-fns | Edge cases (month ends, leap years) |
| Timezone conversion | UTC offset subtraction | @date-fns/tz TZDate | DST handling, IANA zones |
| Slot generation | Complex loops with Date | date-fns addMinutes, format | Clean, testable code |
| State machine | if/else chains | Transition table (existing pattern) | Maintainable, auditable |
| Event emission | Custom pub/sub | @nestjs/event-emitter | Already configured |

**Key insight:** The complexity is in slot generation and double-booking prevention. Use date-fns for reliable date math, database transactions for atomicity, and emit events for Phase 13 notifications.

## Common Pitfalls

### Pitfall 1: Timezone Confusion in Slot Display
**What goes wrong:** Slots show wrong time to users in different timezones
**Why it happens:** Mixing local and UTC times, not specifying timezone
**How to avoid:** Always use Colombia timezone (America/Bogota) for this application. Store UTC in DB, convert for display.
**Warning signs:** Morning slots appearing as previous-day evening slots

### Pitfall 2: Race Condition Double-Booking
**What goes wrong:** Two tenants book same slot simultaneously
**Why it happens:** Check-then-insert without transaction, no database constraint
**How to avoid:** Use Prisma transaction with findFirst + create. Add unique constraint where possible.
**Warning signs:** Multiple PENDING visits for same slot

### Pitfall 3: Availability Change Breaking Existing Visits
**What goes wrong:** Landlord removes availability window that has booked visits
**Why it happens:** No validation when updating availability
**How to avoid:** Check for active visits before allowing availability deletion/modification
**Warning signs:** Visits exist for slots that no longer appear in availability

### Pitfall 4: Slot Duration Mismatch
**What goes wrong:** Generated slots don't align with landlord's configured duration
**Why it happens:** Hardcoded 30-minute slots instead of using slotDuration from config
**How to avoid:** Always read slotDuration from PropertyAvailability record
**Warning signs:** Slots overlap or have gaps

### Pitfall 5: Not Filtering Past Slots
**What goes wrong:** Tenants can book visits in the past
**Why it happens:** No date validation, showing all generated slots
**How to avoid:** Filter slots to only show future times (with buffer, e.g., 2 hours from now)
**Warning signs:** Visits scheduled for times that have already passed

### Pitfall 6: Reschedule Creates Orphan Visits
**What goes wrong:** Original visit stays PENDING when tenant reschedules
**Why it happens:** Not updating original visit status
**How to avoid:** Set original to RESCHEDULED, create new PENDING visit with rescheduledFromId
**Warning signs:** Multiple active visits for same tenant-property

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Pre-generate slots in DB | Generate on-demand | Always preferred | Flexible, no data maintenance |
| moment.js for dates | date-fns v4 with TZ | 2024 | Smaller bundle, tree-shakeable |
| Manual timezone offset | @date-fns/tz TZDate | date-fns v4 | First-class timezone support |
| Check-then-insert | Transaction with constraint | Best practice | Prevents race conditions |
| if/else state checks | State machine pattern | Always preferred | Auditable, maintainable |

**Current best practices:**
- Store availability rules, generate slots on-demand
- Use database transactions for booking creation
- Emit events for cross-module notification integration
- Use TZDate for timezone-aware operations
- State machine for explicit status transitions with role permissions

## Open Questions

Things that couldn't be fully resolved:

1. **Slot Buffer Time**
   - What we know: Tenants shouldn't book slots in the very near future
   - What's unclear: How far in advance minimum? (2 hours? 24 hours?)
   - Recommendation: Start with 2-hour minimum buffer, configurable

2. **Maximum Advance Booking**
   - What we know: Need to limit how far in advance slots are generated
   - What's unclear: 2 weeks? 30 days? Configurable per property?
   - Recommendation: Default 30 days, landlord can customize

3. **Visit Duration After Acceptance**
   - What we know: Slot is blocked when PENDING or ACCEPTED
   - What's unclear: How long after acceptance can tenant cancel freely?
   - Recommendation: Allow cancellation up to 24 hours before, after that mark as "late cancellation"

4. **Automatic Completion**
   - What we know: ACCEPTED visits should become COMPLETED after they happen
   - What's unclear: Automatic via scheduled job, or manual landlord action?
   - Recommendation: Start with manual, add scheduled job later if needed

5. **Multiple Visits Same Property**
   - What we know: Tenant might want to visit property multiple times
   - What's unclear: Should we allow multiple active visits per tenant-property?
   - Recommendation: Allow only one PENDING/ACCEPTED visit per tenant-property
