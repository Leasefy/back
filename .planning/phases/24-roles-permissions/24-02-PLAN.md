# Plan 24-02: Enforcement de Roles de Equipo Propietario

## Goal

Implementar control de acceso para miembros de equipo de propietario (admin, manager, accountant, viewer). Hoy los miembros de equipo acceden a los mismos endpoints que el propietario dueño sin ninguna restricción. Este plan agrega un guard que valida el rol del team member contra una matriz fija de permisos.

### Task 2: Guard `TeamAccessGuard`

**Archivo nuevo:** `src/auth/guards/team-access.guard.ts`

Lógica:
1. Lee metadata `TEAM_PERMISSION_KEY` del handler (ej: `{ resource: 'properties', action: 'edit' }`)
2. Si no hay metadata → pass
3. Obtiene el usuario del request
4. Si `user.role === LANDLORD`:
   a. Verifica si el usuario es el **owner real** del recurso → pass always (es su propio equipo)
   b. Si no es owner → busca `TeamMember` donde `userId = user.id` y status = `accepted`
   c. Lee `teamMember.role` y consulta `TEAM_ROLE_PERMISSIONS[role]`
   d. Verifica `permissions[resource]?.includes(action)`
   e. Si no → `ForbiddenException`
5. Si `user.role === AGENT` → la lógica de agent ya funciona via RolesGuard, pass

**Context scoping (crucial):**
El guard también debe inyectar `request.teamOwnerId` — el ID del propietario dueño. Esto permite que los controllers filtren datos solo del propietario correcto:
- Si es owner directo → `teamOwnerId = user.id`
- Si es team member → `teamOwnerId = teamMember.ownerId`

### Task 3: Decorador `@RequireTeamPermission`

**Archivo nuevo:** `src/auth/decorators/require-team-permission.decorator.ts`

```typescript
import { SetMetadata } from '@nestjs/common';
import { TeamResource, TeamAction } from '../permissions/team-role-permissions.js';

export const TEAM_PERMISSION_KEY = 'team_permission';

export interface TeamPermissionMeta {
  resource: TeamResource;
  action: TeamAction;
}

export const RequireTeamPermission = (resource: TeamResource, action: TeamAction) =>
  SetMetadata(TEAM_PERMISSION_KEY, { resource, action });
```

### Task 4: Actualizar modelo TeamMember — Relación con User

**Archivo:** `prisma/schema.prisma`

Verificar que TeamMember tiene relación con User via `userId` para que el guard pueda resolver la membresía. Actualmente:

```prisma
model TeamMember {
  id         String   @id @default(uuid())
  ownerId    String
  owner      User     @relation("TeamOwner", fields: [ownerId], references: [id])
  userId     String?  // Se llena al aceptar invitación
  name       String?
  email      String
  role       String   // "admin" | "manager" | "accountant" | "viewer"
  status     String   // "pending" | "accepted" | "expired" | "revoked"
  invitedAt  DateTime @default(now())
  acceptedAt DateTime?
}
```

El campo `userId` puede ser null (invitación pendiente). El guard solo busca team members con `status = 'accepted'` y `userId != null`.

**Posible migración:** Agregar índice para optimizar la query del guard:
```sql
CREATE INDEX "TeamMember_userId_status_idx" ON "TeamMember" ("userId", "status") WHERE "status" = 'accepted';
```

### Task 5: Aplicar guard en controllers de landlord

**Controllers a actualizar:**

**5a. `src/landlord/landlord.controller.ts`** (o property controller)
```typescript
@UseGuards(TeamAccessGuard)
@Controller('properties')
export class PropertiesController {
  @Get()
  @RequireTeamPermission('properties', 'view')
  findAll(@Req() req) {
    // Usar req.teamOwnerId para filtrar propiedades
    return this.service.findAll(req.teamOwnerId);
  }

  @Post()
  @RequireTeamPermission('properties', 'create')
  create(...) { ... }

  @Put(':id')
  @RequireTeamPermission('properties', 'edit')
  update(...) { ... }

  @Delete(':id')
  @RequireTeamPermission('properties', 'delete')
  delete(...) { ... }
}
```

**5c. Context scoping en services:**

Cada service que hoy filtra por `userId` debe ser actualizado para filtrar por `teamOwnerId` cuando el usuario es miembro de equipo:

```typescript
// Antes:
findAll(userId: string) {
  return this.prisma.property.findMany({ where: { ownerId: userId } });
}

// Después:
findAll(ownerId: string) {
  return this.prisma.property.findMany({ where: { ownerId } });
}
// El controller pasa req.teamOwnerId en vez de user.id
```

### Task 6: Decorador `@TeamOwner` para extraer ownerId

**Archivo nuevo:** `src/auth/decorators/team-owner.decorator.ts`

Similar a `@CurrentAgency()`, extrae el `teamOwnerId` del request:

```typescript
export const TeamOwner = createParamDecorator(
  (data: undefined, ctx: ExecutionContext) => {
    const request = ctx.switchToHttp().getRequest();
    return request.teamOwnerId; // Inyectado por TeamAccessGuard
  },
);
```

Uso en controllers:
```typescript
@Get()
@RequireTeamPermission('properties', 'view')
findAll(@TeamOwner() ownerId: string) {
  return this.service.findAll(ownerId);
}
```

## Verification

- [ ] El propietario dueño puede hacer todas las operaciones (pass always)
- [ ] Un team member `admin` puede gestionar equipo, propiedades, candidatos, contratos, visitas
- [ ] Un team member `manager` puede gestionar propiedades y candidatos pero NO equipo ni billing
- [ ] Un team member `accountant` solo ve billing, reportes y arriendos (read-only)
- [ ] Un team member `viewer` solo ve propiedades, candidatos, contratos y reportes (read-only)
- [ ] Un team member solo ve datos del propietario que lo invitó (context scoping)
- [ ] Un team member con invitación pendiente (status != accepted) no puede acceder
- [ ] El guard no afecta a usuarios TENANT (no aplica, otro flujo)
