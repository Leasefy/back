Review the evaluation service created in 27-03 and verify/fix all of these:

**A. Pricing Logic Verification (trace each tier path):**

1. **STARTER path:** getActiveSubscription → tier = STARTER → no monthly limit check → deductCredits(userId, 1) → amountPaidCop = 42000 (from evaluationCreditPrice) → creditsDeducted = 1. Verify evaluationCreditPrice is read from planConfig, NOT hardcoded.

2. **PRO path:** getActiveSubscription → tier = PRO → monthly limit check (getMonthlyEvaluationCount) → if >= 30 throw 429 → deductCredits(userId, 1) → amountPaidCop = 21000 → creditsDeducted = 1. After successful evaluation: incrementEvaluationCount. Verify the increment happens AFTER the micro call succeeds (not before — failed evals shouldn't count).

3. **FLEX path:** getActiveSubscription → tier = FLEX → amountPaidCop = 0, creditsDeducted = 0 → NO call to deductCredits. Verify FLEX never touches AgentCreditsService.

4. **No subscription:** getActiveSubscription returns null → throw ForbiddenException (403). Verify the error message is clear.

**B. Error Path Verification:**

1. **Application not found:** `findUniqueOrThrow` should throw → verify it maps to 404 (NestJS default for Prisma NotFoundError). If Prisma throws a generic error, wrap it with `NotFoundException`.

2. **Ownership failure:** Verify the check compares `application.property.landlordId === requestedBy`. Verify the error is `ForbiddenException` (403).

3. **Insufficient credits:** Verify `deductCredits` return value handling — if it returns `false` or count === 0, throw `BadRequestException` with message about buying more credits. Do NOT throw PaymentRequired (402) since 402 is reserved in HTTP spec.

4. **Micro unavailable:** Verify `ServiceUnavailableException` (503) from AgentMicroClient propagates correctly through the controller.

5. **PRO limit exceeded:** Verify the error uses `HttpException('...', 429)` — NOT `TooManyRequestsException` (which may not exist in NestJS). Use `HttpException` directly with status 429.

**C. PRO Monthly Limit Race Condition Hardening:**

The research identifies check-then-increment race condition. Apply optimistic pattern:

```typescript
// In requestEvaluation, AFTER successful micro call and BEFORE returning:
if (chargeResult.tier === 'PRO') {
  const newCount = await this.incrementEvaluationCount(requestedBy);
  // If somehow over limit (race), log warning but don't fail — charge already happened
  if (newCount > 30) {
    this.logger.warn(`PRO user ${requestedBy} exceeded 30 eval limit: ${newCount}`);
  }
}
```

Update `incrementEvaluationCount` to return the new count (return `usage.count` from upsert result).

**D. Idempotency Edge Cases:**

1. Verify that if a FAILED evaluation exists, it's deleted before creating a new one (allows retry).
2. Verify that if a PENDING evaluation exists, `requestEvaluation` returns the existing runId WITHOUT re-charging.
3. Verify that if a COMPLETED evaluation exists, `requestEvaluation` returns the existing result WITHOUT re-charging.

**E. GET /evaluations/:applicationId/result Hardening:**

1. If evaluation doesn't exist → `NotFoundException`
2. If user doesn't own the application → `ForbiddenException`
3. If status is PENDING and micro poll returns a result → update to COMPLETED in DB and return
4. If status is PENDING and micro poll returns null → return PENDING status
5. If micro poll throws → log the error, return PENDING status (don't crash the GET endpoint)

Wrap the pollResult call in a try/catch in getResult — if polling fails, the user should still see the current PENDING status, not a 500 error.

**F. Swagger Documentation:**

Ensure the controller has these `@ApiResponse` decorators on POST:
- 202: Evaluación iniciada, runId devuelto
- 400: Créditos insuficientes
- 403: Sin suscripción activa o sin acceso a la solicitud
- 404: Solicitud no encontrada
- 429: Límite PRO de 30 evaluaciones/mes alcanzado
- 503: Microservicio de agentes no disponible

And on GET:
- 200: Estado actual de la evaluación
- 403: Sin acceso a la solicitud
- 404: No se encontró evaluación
