# Phase 2.2 Research: Inmobiliaria Backend

## Overview

The frontend has a comprehensive inmobiliaria (real estate agency) module with **21 pages**, **83+ components**, and **7 mock data files** totaling ~4,000+ lines of TypeScript types and mock records. The backend currently only has a basic AGENT role with PropertyAccess model (Phase 2.1). This phase bridges that gap.

## Current Backend State

**Existing (Phase 2.1):**
- `Role.AGENT` in Prisma enum
- `PropertyAccess` model (agentId, propertyId, grantedById, isActive)
- `AgentsController`: 2 endpoints (`GET /agents/me/properties`, `GET /agents/me/properties/:propertyId/landlord`)
- `PropertyAccessService`: canAccessProperty(), assignAgent(), removeAgent()
- No Agency entity, no agency-specific modules

**What's Missing:**
- Agency/Inmobiliaria entity (organization that groups agents)
- Propietario (property owner) management
- Consignacion (property consignment agreements)
- Pipeline (sales funnel with 10 stages)
- Cobros (monthly rent collection tracking)
- Dispersiones (owner disbursement processing)
- Mantenimiento (maintenance request management)
- Renovaciones (lease renewal workflow)
- Actas de Entrega (property delivery/return acts)
- Document templates and generation
- Agency configuration and team management
- Reports, analytics, and dashboard

## Architectural Decisions

### 1. Agency as Organization
Create an `Agency` model that groups users. The existing `AGENT` role becomes a user within an agency. This enables multi-tenancy for agencies.

### 2. Propietario as Separate Entity
Propietarios are NOT the same as User with Role.LANDLORD. In the agency context, propietarios are clients of the agency - they may or may not have a platform account. The agency manages their properties on their behalf.

### 3. Consignacion vs Property
A Consignacion wraps an existing Property with agency-specific data (commission %, assigned agent, contract dates). The underlying Property model stays unchanged.

### 4. Cobro Generation
Cobros are generated monthly from active consignaciones with leases. A scheduled job or manual trigger creates them.

### 5. Dispersion as Aggregation
Dispersiones aggregate cobros by propietario per month, deduct commissions, and track bank transfer status.

### 6. Module Structure
All inmobiliaria endpoints live under a single `InmobiliariaModule` with sub-modules:
- `InmobiliariaModule` (root, under `/inmobiliaria`)
  - `PropietariosModule`
  - `ConsignacionesModule`
  - `PipelineModule`
  - `CobrosModule`
  - `DispersionesModule`
  - `MantenimientoModule`
  - `RenovacionesModule`
  - `InmobiliariaDocumentsModule`
  - `InmobiliariaReportsModule`
  - `InmobiliariaAnalyticsModule`
  - `InmobiliariaConfigModule`

### 7. Authorization
Agency endpoints require the user to be a member of an agency (via AgencyUser). Role checks within agency: admin, agente, contador, viewer.
